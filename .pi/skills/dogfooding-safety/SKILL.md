# Dogfooding Safety

Use this skill when work touches the root `.devcontainer/`, `.devcontainer/custom/`, `overlays/**`, `templates/**`, `features/**`, `tool/schema/**`, or generated outputs such as `docs/overlays.md`, `tool/schema/superposition.schema.json`, and `dist/`.

## When to use this skill

- Reviewing or changing files under root `.devcontainer/`
- Reviewing or changing `overlays/**`, `templates/**`, or `features/**`
- Reviewing or changing `tool/schema/**`
- Working with generated outputs: `dist/`, `docs/overlays.md`, `tool/schema/superposition.schema.json`
- Investigating or fixing reproducibility, regen, or doctor failures
- Running or verifying `init -- regen`, `init -- doctor`, `docs:generate`, or `schema:generate` flows

## Read first

Before making any changes, read these in order:

1. `AGENTS.md` — authoritative project contributor rules; owns the Do Not Edit list and non-obvious path/ESM rules
2. `docs/foundation.md` — engineering boundary authority; defines generated vs. authored surfaces
3. `docs/definition-of-done.md` — validation gates including the reproducibility and doctor requirements
4. `docs/adr/adr001-project-file-first-replay-and-regeneration.md` — project-file-first intent, generated artifact authority, and git-safety rules
5. `CONTRIBUTING.md` — contributor-facing guidance on dogfooded output
6. `.github/instructions/dogfooding.instructions.md` — dogfooding-specific instructions for this repo
7. The owning source files or governing specs for the specific generated artifact being changed

## Required workflow

### 1. Treat generated outputs as source-owned artifacts

These files are **outputs**, not the default edit surface:

- `dist/` — compiled TypeScript output
- `docs/overlays.md` — regenerated by `npm run docs:generate`
- `tool/schema/superposition.schema.json` — regenerated by `npm run schema:generate`
- Root `.devcontainer/` — the dogfooded generated devcontainer for this repository

Fixes must be routed back to the owning source files:

- Build errors → fix TypeScript source, then rebuild
- Wrong overlay docs → fix overlay source files (`overlays/<id>/overlay.yml`, etc.), then `npm run docs:generate`
- Wrong schema output → fix `tool/schema/types.ts` or overlay source, then `npm run schema:generate`
- Wrong root devcontainer content → identify the responsible overlay, template, or generator logic and fix it at source, then `npm run init -- regen`

Authority: `AGENTS.md` ("Do not edit" section), `docs/adr/adr001-project-file-first-replay-and-regeneration.md`

### 2. Treat the root `.devcontainer/` as dogfooded generated output

The root `.devcontainer/` directory reflects the output of running this project's own tool against itself. This means:

- Duplicate or verbose generated settings are **not defects** by default — they reflect composition of overlays
- Do not treat cosmetic deduplication as a valid reason to edit root `.devcontainer/` files directly
- When reviewing root `.devcontainer/` changes, evaluate runtime correctness, generation accuracy, and reproducibility — not cosmetic tidiness
- If the root devcontainer appears to contain errors, investigate whether the source (overlays, templates, generator logic) is the true origin before touching the output

Authority: `.github/instructions/dogfooding.instructions.md`, `CONTRIBUTING.md`

### 3. Treat `.devcontainer/custom/` as a separate review surface

`.devcontainer/custom/` is intentional override space:

- Files there are **authored**, not generated — apply normal code-review scrutiny
- Changes there do not require `init -- regen` unless they affect how regen composes output
- Evaluate `.devcontainer/custom/` changes for correctness, necessity, and whether they should instead be expressed as an overlay

### 4. Route fixes to the true source

When you identify a problem in a generated artifact, trace it to the owning source before writing any code:

| Symptom                                         | True source to fix                                                          |
| ----------------------------------------------- | --------------------------------------------------------------------------- |
| Wrong overlay doc content                       | `overlays/<id>/overlay.yml` or `overlays/<id>/README.md`                    |
| Wrong schema validation                         | `tool/schema/types.ts` or overlay `overlay.yml` parameter definitions       |
| Wrong compose or devcontainer generated content | `overlays/<id>/devcontainer.patch.json`, `overlays/<id>/docker-compose.yml` |
| Wrong feature or template output                | `features/<id>/` or `templates/`                                            |
| Wrong CLI behavior                              | TypeScript source in `tool/commands/`, `tool/cli/`, `scripts/`              |

Do not patch generated root output to remove the visible symptom — that fix will be overwritten on the next regen and masks the real problem.

Authority: `docs/adr/adr001-project-file-first-replay-and-regeneration.md`, `docs/foundation.md`

### 5. Run the required regeneration and reproducibility checks

When the change affects generated output:

| Trigger                                                                                    | Command                                                      |
| ------------------------------------------------------------------------------------------ | ------------------------------------------------------------ |
| Overlay source files changed                                                               | `npm run docs:generate`                                      |
| Overlay source or schema types changed                                                     | `npm run schema:generate`                                    |
| User-visible or tooling changes affecting generated devcontainer or other generated output | `npm run init -- regen`                                      |
| Any generated-output workflow before merge                                                 | `npm run init -- doctor` — no reproducibility errors allowed |

Run these checks **before** committing. If `init -- doctor` reports a reproducibility failure, trace it to source and fix before merging.

Authority: `AGENTS.md` (Commands section), `docs/definition-of-done.md`

### 6. Preserve project-file-first and git-safety boundaries

- Canonical intent lives in `superposition.yml` / `.superposition.yml` — the source that drives replay, regen, and doctor
- `superposition.json` is compatibility/reproducibility output — not the file to hand-edit when a project file exists
- Git-index mutation (staging, unstaging, tracking changes) remains **manual contributor guidance**, not automatic contributor automation
- Do not add automation that silently mutates tracked files or git state without an approved spec authorizing it

Authority: `docs/adr/adr001-project-file-first-replay-and-regeneration.md`

## Do not do this

- Edit `dist/` directly — fix TypeScript source and rebuild
- Make routine direct edits to generated root `.devcontainer/` files — fix the source and regen
- Treat harmless generated duplication or verbosity as a defect requiring manual cleanup
- Ship generator-affecting changes without running `npm run init -- regen` and `npm run init -- doctor`
- Patch a generated artifact's output to hide a symptom without fixing the owning source

## Escalate when

- A generated artifact appears wrong but the owning source module or overlay is not clear — escalate rather than guessing
- A requested fix would normalize direct edits to generated output as a standard workflow pattern
- Reproducibility rules appear to conflict with the requested contributor workflow
- Work conflicts with `AGENTS.md`, `docs/foundation.md`, `docs/definition-of-done.md`, ADR 001, or an approved feature spec — escalate; do not invent a local exception

## Validation

Select validation by change type:

| Change type                              | Required validation                                                                    |
| ---------------------------------------- | -------------------------------------------------------------------------------------- |
| Formatting-affecting file changes        | `npm run lint:fix` then `npm run lint`                                                 |
| Any shipped change                       | `npm run lint`                                                                         |
| Targeted source change                   | Targeted tests for the changed module at minimum                                       |
| Broad source refactor                    | `npm test`                                                                             |
| CLI source change                        | `npm run build` + verify compiled output behavior                                      |
| Overlay source change                    | `npm run docs:generate` (and `npm run schema:generate` if parameters or types changed) |
| Schema-type change                       | `npm run schema:generate`                                                              |
| Generator, template, or feature change   | `npm run init -- regen`                                                                |
| Any generated-output change before merge | `npm run init -- doctor` — no reproducibility errors                                   |

Full command reference: `AGENTS.md` → Commands section.
